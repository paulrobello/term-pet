# Findings: term-pet integration vs. standalone screen-watcher

Written 2026-04-23. Based on reading `~/projects/term-pet/` at current HEAD and
looking up Claude Code / Anthropic SDK docs for the LLM-commentary path. Treat
model/subscription specifics as "true as of April 2026" — Anthropic moves fast,
re-check before committing.

---

## Summary recommendation

- **Path A (integrate into term-pet)** is cleaner but semantically lossy: term-pet's
  abstraction is a 3-state terminal pet with 6 blink-oriented frames; our pet is a
  6-state spatial pet with walk/fall/land. You will either fight the mismatch or
  redefine one of the two.
- **Path B (standalone screen-watcher + LLM commentary)** has no architectural
  friction, but the "use Claude subscription" expectation doesn't actually hold up:
  programmatic image+prompt calls require the pay-per-token Anthropic API (API
  key). A Claude Max subscription cannot currently fund those calls through any
  documented path.
- **A hybrid** — our Swift pet remains its own app, but delegates LLM
  commentary to term-pet's existing watchdog+LLM pipeline via a local socket or
  file — buys you 80% of both paths with the least new code. Worth considering.

## term-pet structure (as-is)

- Python 3.13+, Typer CLI (`tpet`), Hatchling build, config at `~/.config/tpet/`.
- Animation states: `IDLE`, `REACTING`, `SLEEPING` (three). Frames are a 2×3
  sprite sheet: `0-1` idle-open, `2-3` blink, `4` surprised/reacting, `5`
  sleeping. Source: `src/tpet/animation/engine.py`.
- Renderer is a `Protocol` at `src/tpet/renderer/protocol.py`:
  ```python
  class Renderer(Protocol):
      def render(self, live: Live, pet: PetProfile, frame_idx: int,
                 current_comment: str | None,
                 frame_changed: bool, comment_changed: bool) -> None: ...
  ```
  Two implementations today: `AsciiRenderer` (Rich layout, ASCII text) and
  `HalfblockRenderer` (ANSI half-block from PNG sprites). Both render **into a
  Rich `Live` terminal object**.
- Art assets live in `~/.config/tpet/art/{pet_name}_frame_{N}.png` and
  `{pet_name}_sprite.png` (sheet). Generated by Gemini, per-pet.
- Commentary: a watchdog observes Claude Code IDE session logs and drives
  in-character comments.

## Path A — Integrate as a term-pet display mode

### What "just another display" would mean

term-pet's `Renderer` protocol is inherently tied to the `Live` terminal
object. A desktop-window renderer won't fit literally — you'd either:

1. **Add a new `DesktopRenderer` class** that ignores the `Live` argument and
   instead signals a long-running Swift process over a local socket /
   `stdin`: `{ "frame_idx": 4, "comment": "oh no" }`. Swift owns the
   actual window; Python owns the state machine. Needs matching entry in
   `src/tpet/app.py:_build_renderer()`.
2. **Or** have the Swift process read term-pet's art dir + a shared comment
   file directly — no renderer changes on the Python side, but the Swift
   process has to poll or watch the filesystem. Hackier; losing
   frame-change events.

Option 1 is the right call if you want this to feel like a first-class
term-pet display mode.

### State / frame mismatch to resolve

| Our state (Swift) | term-pet state  | Frame                   |
|-------------------|-----------------|-------------------------|
| `idle`            | `IDLE`          | 0 or 1 (blink from 2/3) |
| `walking`         | **none**        | Interpolate from idle?  |
| `sleeping`        | `SLEEPING`      | 5                       |
| `dragging`        | **none**        | 4 (react) as a proxy?   |
| `falling`         | **none**        | 4 (react)               |
| `landed`          | `REACTING`      | 4 + stars overlay       |

`walking`, `dragging`, `falling`, `landed` have no natural home in term-pet's
state machine. Two honest choices:

- **Shrink the Swift pet** to only 3 states (idle/react/sleep) when run
  under term-pet. Drop window-perching and drag physics. The pet becomes a
  fixed-position graphical version of the terminal pet. Simple, lossy.
- **Grow term-pet** to add movement states. Bigger change — touches the
  animator, the sprite sheet layout, the art-generation prompts in
  `src/tpet/art/`, and anything that hardcodes `FRAME_COUNT_CURRENT = 6`.
  "Generating more artworks" in your original message maps here.

### Art generation TODO (if growing term-pet)

1. Extend `FrameLayout` to include `walk_a`, `walk_b`, `fall`, `landed`.
   Probably 3×3 or 4×2 sheet.
2. Update the Gemini prompt template to generate the extra poses
   (`src/tpet/art/` — look at the sprite-sheet prompt and add guidance:
   "mid-walk stride, airborne, stunned on ground with X eyes").
3. Update slicing/validation to the new grid size.
4. Our Swift loader already normalizes any PNG via
   `Sprites.compose(sourceImage:)` (scale so max dim = 64, anchor
   center-bottom). That part carries over.
5. Map term-pet frame indices → Swift `SpriteFrame` in a small table.

### TODOs — Path A (Swift-as-display path, option 1 above)

1. Decide the IPC surface. Minimum: a Unix domain socket where Python emits
   line-delimited JSON `{"frame": int, "comment": string|null}` on every
   `frame_changed` or `comment_changed`. Swift reads the socket on a
   background `DispatchQueue` and posts to the main actor.
2. In `Sheep/`, add a `TermPetClient` module: connects to
   `~/.config/tpet/display.sock`, decodes JSON, pushes frame + comment into
   `SheepController`.
3. Replace the Swift state machine with a pure "render the frame Python
   told me" mode when term-pet is driving. Keep the existing autonomous
   state machine as a fallback when not connected.
4. Add speech-bubble rendering in Swift (NSTextField in a rounded
   NSVisualEffectView, pinned above the sheep). This is new UI work either
   way — see Path B.
5. In `term-pet/`, add `DesktopRenderer(Renderer)` and wire it into
   `_build_renderer()` under a new `art_mode` value like `"desktop"`.
6. Terminal launch: `tpet --art-mode desktop` spawns the Swift app
   (`subprocess.Popen`) after opening the socket. Or require the Swift
   app to be already running and term-pet just connects.
7. Decide pet-name propagation: Swift needs to know which pet to load
   sprites for, from `~/.config/tpet/art/`. Simplest: Python sends pet name
   as the first message on the socket.
8. Art generation — only if you also grow the sprite grid (see above).

### Feasibility — Path A

- **Engineering:** medium. Mostly plumbing. 1–2 focused days for the
  IPC-based display (option 1) without growing the sprite set; add 1–2 days
  for the art pipeline extension.
- **Design:** the 3-state vs 6-state mismatch is the honest problem. If you
  accept term-pet's model, drop movement features. If you grow term-pet's
  model, it's a real change to term-pet's core.
- **Cross-platform:** term-pet claims Linux/Windows/Mac. The Swift display
  is Mac-only. So this integration effectively gives term-pet a Mac-only
  extra display mode. Fine — that's what desktop renderers do.

## Path B — Standalone screen-watcher with LLM commentary

### Components

1. **Screen capture** on macOS: `ScreenCaptureKit` (macOS 12.3+, async
   `SCStream` or one-shot `SCScreenshotManager.captureImage`). Requires
   the Screen Recording permission — first-run prompt, user must approve
   in System Settings → Privacy & Security → Screen Recording.
2. **Rate limiting & trigger**: don't sample every frame. Default to
   once every 20–60 s, or on a "user is idle" heuristic, or on an
   explicit poke (click the sheep). Otherwise cost and noise explode.
3. **LLM call**: multimodal request with the screenshot (downscaled —
   send at 1024×640 or so; model tokens scale with image size) + a short
   prompt like *"In one line, as a sheep, comment on what the user is
   doing. Keep it playful, max 15 words."*
4. **Speech bubble UI**: same component needed in Path A. NSTextField in
   a rounded panel, anchored above the sheep's current window position,
   auto-dismiss after N seconds.
5. **Privacy controls**: on/off toggle in the status-bar menu. Blur or
   exclude specified apps (Password managers, Messages, Mail). Make the
   default something the user has to opt into.

### "How do I use my Claude subscription?" — reality

Short answer: **you probably can't.** Details:

- **Pay-per-token Anthropic API (API key)** is the only cleanly documented
  path for a programmatic `image + prompt → text` call. Auth via
  `ANTHROPIC_API_KEY` env var, SDKs for Python/TypeScript/Swift. This is
  what the screen-watcher will actually use.
- **Claude Code CLI (`claude -p`)** supports non-interactive "ask once,
  exit" calls and *can* be authenticated with a Claude Pro/Max
  subscription (OAuth login via `claude` + browser). Max's included
  credits then fund those calls. **However:** image inputs to `claude -p`
  are not documented as a supported flag today. If that changes, a Swift
  shell-out to `claude -p --image screenshot.png "comment as a sheep"`
  becomes the cleanest subscription-funded path.
- **Claude Agent SDK (Python/TypeScript)** is the programmatic cousin of
  Claude Code. As of the docs I read, it authenticates via API key, not
  subscription OAuth, and image inputs to `query()` aren't documented.
  Not the right tool for this use case today.
- **What Anthropic clearly does not support:** using a Pro/Max
  subscription to power calls made through the raw messages API. The
  subscription and Console API are billed separately.

**Practical recommendation:** use the Anthropic API with an API key.
At Haiku 4.5 rates (~$1 / MTok input, $5 / MTok output), a 1024×640
screenshot plus short prompt and a 30-token response is ≈ $0.003–0.005
per call. One call per minute for an 8-hour day ≈ $1.40–2.40. Not free,
not ruinous. Make the sampling interval user-configurable.

If the user strictly refuses pay-per-token, the only documented
subscription-funded fallback is manual: run `claude` interactively in
a side terminal and paste screenshots by hand. Not useful for live
commentary. Re-check Claude Code and Agent SDK release notes before
locking this in — image support on `claude -p` is the kind of feature
that lands quietly.

### TODOs — Path B

1. Add Screen Recording permission flow. Check `CGPreflightScreenCaptureAccess()`,
   `CGRequestScreenCaptureAccess()` for the older API; for ScreenCaptureKit,
   detect permission by trying a capture and handling the error.
2. Add `ScreenSampler`: async actor that captures the current screen at an
   interval, downscales to ≤1024 wide via `CIFilter` or `CGImage` resize,
   encodes as JPEG (smaller than PNG for screenshots), returns `Data`.
3. Add `CommentaryClient`: talks to the Anthropic API via `URLSession`. No
   SDK dependency needed — the messages endpoint is a plain POST. Reads
   `ANTHROPIC_API_KEY` from env var or Keychain.
4. Add `SpeechBubbleWindow`: a second NSWindow next to the sheep, rounded
   rect with text, fade in/out. This is the same UI needed for Path A;
   build it once.
5. Add a status-bar menu item: "Enable screen commentary" (default off),
   "Sampling interval: 30 s / 60 s / manual", "Quit".
6. Add a manual trigger: double-click on the sheep → immediate commentary
   (overrides the interval). Cheaper than always-on.
7. Cost + privacy hygiene: redact specified windows by comparing
   `CGWindowListCopyWindowInfo` owners against an exclusion list and
   masking those regions to black before sending.
8. Pick a default model. Haiku 4.5 for speed+cost on quick comments; upgrade
   to Sonnet 4.6 only when the user asks for more clever quips. Expose in
   settings later.
9. Prompt engineering: keep max_tokens low (~60), temperature moderate,
   system prompt establishing the sheep's persona and a hard length cap.

### Feasibility — Path B

- **Engineering:** medium-high. ScreenCaptureKit has sharp edges around
  permissions and async streams; the Anthropic POST is trivial; the
  speech-bubble window is routine. 3–5 focused days to a polished first
  version.
- **UX / ethics:** the elephant in the room is that the sheep will see
  your password manager, your DMs, and your tax return. Default-off,
  explicit opt-in, per-app exclusions, and a visible "commenting…"
  indicator are not optional. This is as much a design problem as an
  engineering one.
- **Cost:** manageable at Haiku with sensible intervals (~$1–2/day for
  heavy use). Manual-trigger mode is nearly free.
- **Latency:** expect 1.5–3 s from capture to bubble. Fine for periodic
  commentary, unacceptable for anything real-time — don't promise it.

## Hybrid worth considering

Keep Swift pet standalone (Path B), but **route commentary generation
through term-pet's existing pipeline** instead of re-doing LLM prompting.
term-pet already watches Claude Code IDE sessions, owns the pet's persona,
and issues comments. Have Swift listen for comments from term-pet over a
socket or `~/.config/tpet/comments.ndjson` file and render them in the
speech bubble.

- Avoids re-implementing prompt design, persona, model choice in Swift.
- Keeps term-pet's cross-platform terminal mode intact.
- Your screen-watcher idea becomes an optional second comment source
  feeding the same pipeline.

This is what I'd build if it were mine: Path A-lite (share art + comment
bus only, skip the renderer-as-IPC contortion), Path B for screen
commentary, both feeding a shared bubble.
